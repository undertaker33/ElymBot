## 模块职责

本模块当前覆盖：

- `:feature:plugin:api`：插件安装、catalog、配置、状态、治理、运行时观察、workspace path、host capability、V2 message / LLM / tool contract 和模型 contract。
- `:feature:plugin:data`：Room-backed plugin repository、catalog store、config snapshot、host config resolver、plugin/session state store、package validation、storage path 和 uninstall cleanup。
- `:feature:plugin:presentation`：插件 local / market workspace、详情、管理、配置、日志、触发器、workspace file 管理和 schema renderer。
- `:feature:plugin:runtime`：插件安装、catalog sync、runtime observation、V1 frozen dispatch boundary、V2 QuickJS runtime、message dispatch、LLM pipeline、tool loop、host capability、runtime logs、failure governance、ToolSource 和 MCP integration。
- `app-integration` wiring：通过 Hilt 提供 plugin data/runtime/provisioning/host capability/observation/services，不在 `app` 内手写 production plugin graph。
- `app` UI 入口：通过 navigation 接入 plugin root、detail、manager、market detail、workspace、logs、triggers 和 config 页面。

## 非职责

- Cron runtime 的 schedule persistence、coordinator 和 run-now 执行不在本模块完成态；这里只确认 plugin active capability 如何通过 cron port 消费它们。
- Resource Center 的数据真源、设置页和备份入口不在本模块完成态；这里只确认 ToolSource context 从 `ResourceCenterPort` 投影读取 MCP / skill / web search 配置。
- QQ / App Chat 的消息入口不在本模块重新完成；这里只确认它们通过 plugin API/capability port 消费 plugin runtime。
- Provider、Config、Bot、Persona 的仓储和 profile 规则已在第 6 模块确认；本模块只记录 plugin host config 和 ToolSource context 对这些 port 的依赖。
- STT/TTS、voiceasset 和 audio runtime 仍待后续 `voiceasset-audio` 模块确认。

## 当前 Gradle 与模块边界

`settings.gradle.kts` 当前声明：

- `:feature:plugin:api`
- `:feature:plugin:data`
- `:feature:plugin:impl`
- `:feature:plugin:presentation`
- `:feature:plugin:runtime`

依赖事实：

- `:feature:plugin:api` 依赖 `core:runtime-context`、chat/conversation/persona API、coroutines、JSON 和 `javax.inject`。
- `:feature:plugin:data` 依赖 core common/db/logging 与 plugin API，并提供 Hilt-ready repository/config/state/cleanup 实现。
- `:feature:plugin:presentation` 启用 Compose + Hilt + KSP，依赖 core common/runtime/logging/ui、plugin API/data 和 Compose Material。
- `:feature:plugin:runtime` 依赖 core runtime/context/search/tool、download API、config/conversation/cron/persona/resource API、plugin API/data、QuickJS、OkHttp、Hilt 和 JSON。
- `:feature:plugin:impl` 当前没有生产 Kotlin/Java 文件，是聚合壳，`api` 暴露 plugin API/data/presentation/runtime，并复用 app resources；不要把它写成生产 owner。
- `app-integration` 以 `api(project(":feature:plugin:api"))` 暴露 plugin API，以 `implementation` 接 `:feature:plugin:data` 和 `:feature:plugin:runtime`。
- `app/build.gradle.kts` 生产只直接 `implementation(project(":feature:plugin:presentation"))`；plugin data/runtime 通过 `app-integration` 和 Hilt 进入生产图。
- root `build.gradle.kts` 当前有 `modulePluginBuild` / `modulePluginCheck` 模块组，覆盖 plugin API/data/impl/runtime/presentation。

## Hilt 与启动主线

当前生产主线：

1. `AppBootstrapper -> AppStartupRunner`
2. `AppStartupRunner` 在 `ReferenceGuardStartupChain` 后运行 `PluginRuntimeObservationStartupChain`
3. `PluginRuntimeObservationStartupChain` 调 `PluginRuntimeObservationPort.startObserving()`
4. `PluginRuntimeObservationPortAdapter` 监听 `PluginStateRepositoryPort.records`，通过 `PluginV2RuntimeLoader.sync(...)` 和 `PluginV2LifecycleManager.signalReady(...)` 同步启用插件
5. `PluginRuntimeModule` 提供 plugin repository ports、runtime log bus、governance、failure guard、scheduler、V2 active store、dispatch engine、lifecycle manager 和 runtime loader
6. `PluginProvisioningModule` 提供 catalog fetcher、remote package downloader、installer、catalog synchronizer、repository subscription manager 和 install intent handler
7. `PluginHostCapabilityModule` 提供 `PluginExecutionHostResolver`、`PluginExecutionHostOperations`、`ExternalPluginHostActionExecutor` 和 `PluginHostCapabilityGatewayFactory`
8. `PluginRuntimeServicesModule` 提供 `PluginHostCapabilityGateway`、`PluginExecutionEngine`、`ExternalPluginRuntimeCatalog`、`EngineBackedAppChatPluginRuntime`、`PluginV2MessageDispatchPort`、`FutureToolSourceContextResolver`、`FutureToolSourceRegistry` 和 `RuntimeLlmOrchestratorPort`

`PluginDataWiringFactory` 是当前 app-integration 的精确 factory seam，用来从 Room DAO、`Context.filesDir`、runtime loader provider 和 Hilt-owned services 组装 plugin repository、config/state store、runtime loader、installer、package validation 和 host operations。

不要恢复 `PluginRuntime*Provider.installFromHilt(...)`、手写 runtime subgraph、static repository facade 或 `PluginRuntimeDependencyBridge` 作为生产 DI。

## Data / config / state / cleanup

当前数据真源：

- `FeaturePluginRepository` 实现 `PluginRepositoryPort` 及 install/catalog/config/state 子 port，持有 `PluginInstallAggregateDao` 和 `PluginCatalogDao`。
- `RoomPluginConfigStorage` 负责读取/保存 plugin config snapshot，并解析已安装插件的 static config schema / settings schema 路径。
- `DefaultPluginHostConfigResolver` 合并 static defaults、settings schema defaults、persisted core config、persisted extension config 和 workspace snapshot，向 host API 提供 `mergedSettings`。
- `RoomPluginStateStore` 持久化 `plugin` 与 `session` 两类 scope，key 会 trim 且不能为空，单值 JSON 大小限制为 64 KB。
- `DefaultPluginDataCleanupService` 卸载时清理 runtime artifact、package/extracted dir；只有 `PluginUninstallPolicy.REMOVE_DATA` 才删除 config snapshot、state entries 和 private workspace。
- `PluginStoragePaths.fromFilesDir(appContext.filesDir)` 当前使用 `files/plugins/packages`、`files/plugins/extracted` 和 `files/plugins/private` 作为插件包、解压目录和私有 workspace 根。

`FeaturePluginRepository` 不再是配置、状态和卸载文件清理的唯一实现点；这些能力已经拆到明确 port/service。

## Runtime / V1 / V2

当前 runtime 边界：

- `PluginRuntimeFacade` 是很薄的 `PluginRuntimePort` facade，委托 `PluginV2RuntimeLoader`、`PluginV2ActiveRuntimeStore` 和 `PluginRuntimeLogBus`。
- `PluginV1DispatchAdapter` 是 frozen V1 boundary，旧 `PluginV1LegacyAdapter.kt` 已不应作为生产文件存在。
- `PluginExecutionEngine` 执行 runtime plugin、failure guard、scheduler 和 result merge。
- `ExternalPluginRuntimeCatalog` 通过 `PluginRepositoryStatePort` 读取当前启用插件，V2 record 不进入 legacy external runtime catalog。
- `PluginV2RuntimeLoader` 负责 eligible record sync/load/reload/unload、QuickJS bootstrap、registry compile、activation diagnostics 和 active store 更新。
- `PluginV2DispatchEngine` 负责 message stage、command、regex、LLM hook dispatch 和 propagation lifecycle。
- `PluginV2BootstrapHostApi` 向 QuickJS bootstrap 暴露 message/command/regex/lifecycle/LLM/tool registration、log、settings 和 plugin/session storage。
- `DefaultRuntimeLlmOrchestrator` 是共享 LLM pipeline 的当前实现，经 `RuntimeLlmOrchestratorPort` 注入给 App Chat、QQ 和 Cron 相关 runtime。

V1 residual triggers 只能留在 compat-only 位置；App Chat、QQ 和 LLM 在线主线不得回到 `ExternalPluginTriggerPolicy` 或旧 legacy adapter 命名。

## Plugin install / catalog / provisioning

当前安装与市场主线：

- `PluginPackageValidator` 要求 `manifest.json`，并支持 `android-plugin.json` package contract；legacy `android-execution.json` 只作为校验 issue。
- `PluginPackageContract.SUPPORTED_PROTOCOL_VERSION` 由 `PluginProvisioningModule.provideSupportedPluginProtocolVersion()` 暴露。
- `PluginInstaller` 负责本地包、remote package、repository version 和 upgrade install；remote download 通过 `RemotePluginPackageDownloader`。
- 生产 remote download 走 `DownloadManagerRemotePluginPackageDownloader`，download owner 使用 `plugin:<packageUrl sha256>` task key。
- `PluginCatalogSynchronizer`、`PluginRepositorySubscriptionManager` 和 `PluginInstallIntentHandler` 由 `PluginProvisioningModule` 提供。
- app 层 deep link `astrbot://plugin` 和 `https://astrbot.local/plugin` 已在 Manifest 中存在，但安装 intent 处理属于 plugin provisioning 主线，不是 app 手写安装逻辑。

## Host capability / execution host

当前生产边界：

- `PluginExecutionHostResolver` 是 host snapshot、context injection、host builtin tools 和 host tool execution 的抽象入口。
- `DefaultPluginExecutionHostResolver` 通过 `DefaultPluginExecutionHostOperations` 实现。
- `PluginHostCapabilityGatewayFactory` 从 Hilt-owned resolver 与 executor 创建 contextual gateway。
- `DefaultPluginHostCapabilityGateway` 负责 host action execution、context injection、host builtin tools、tool allow check。
- `ExternalPluginHostActionExecutor` 通过 permission grant / whitelist / failure guard 执行 host action。

`PluginExecutionHostApi` 仍是 runtime compat/test residue，不是生产 mainline；新代码不能把 `PluginExecutionHostApi.resolve/inject/registerHostBuiltinTools/executeHostBuiltinTool` 写回生产热路径。

## ToolSource / MCP / Skill / Web Search / Active capability

当前 ToolSource 主线：

- `FutureToolSourceRegistry` 是 Hilt `@Singleton` registry，支持 contract `ToolDescriptor` 与 legacy plugin descriptor 两条读取路径。
- `PortBackedFutureToolSourceContextResolver` 通过 `ConfigRepositoryPort + ResourceCenterPort` 构造 `ToolSourceContext`。
- `PluginToolSourceBridge` 负责 descriptor cache。
- 默认 provider 包括 `McpToolSourceProvider`、`SkillToolSourceProvider`、`ActiveCapabilityToolSourceProvider`、`ContextStrategyToolSourceProvider` 和 `WebSearchToolSourceProvider`。

边界细节：

- `McpToolSourceProvider` 只支持 remote `streamable_http` MCP transport；stdio subprocess 和 legacy SSE 不能写成当前可用路径。
- `SkillToolSourceProvider` 只处理 tool skill projection；prompt skill 仍进入 PromptAssembler，不要把所有 skill 都写成工具。
- `WebSearchToolSourceProvider` 只在 `ToolSourceContext.webSearchEnabled` 为 true 时暴露工具，并通过 `UnifiedSearchPort` 搜索。
- `ActiveCapabilityToolSourceProvider` 暴露 future task 管理工具，但 ingress trigger 为 `SCHEDULED_TASK` 时隐藏，避免计划任务执行时再次创建或管理计划任务。
- `ActiveCapabilityRuntimeFacade` 负责创建、删除、暂停、恢复、更新、run-now 和列出 future task；缺失 target context 返回 structured error `missing_context`。
- `ContextStrategyToolSourceProvider` 当前提供 `ctx.compress` 类 context strategy 工具。

## Presentation / UI 入口

当前 UI 主线：

- `PluginViewModel` 是 `@HiltViewModel`，依赖 `PluginViewModelBindings`、`PluginPresentationController`、`PluginPackageValidationPort` 和 `SavedStateHandle`。
- `DefaultPluginViewModelBindings` 拆成 `PluginMarketBindings`、`PluginConfigBindings`、`PluginWorkspaceBindings`、`PluginGovernanceBindings` 和 management bindings。
- `PluginPresentationController` 只通过 `PluginManagementBindings` 调 enable / disable / uninstall / recover / suspend / refresh catalog，不在 ViewModel 内手写 repository/runtime 组装。
- `PluginViewModelBindingsModule` 将 state flows 从 `PluginStateRepositoryPort` 暴露给 UI，并绑定 `DefaultPluginViewModelBindings`。
- `PluginScreen` 处理 local / market workspace；`PluginDetailScreen`、`PluginConfigScreen`、`PluginRuntimeLogScreen`、`PluginTriggerManagementScreen`、`PluginWorkspaceScreen` 覆盖详情、配置、日志、触发器和 workspace。
- `AppDestinations` 与 `AstrBotAppScaffoldParts` 接入 plugin routes：`plugins`、`plugins/detail/{pluginId}`、`plugins/manager`、`plugins/market/detail/{pluginId}`、`plugins/detail/{pluginId}/workspace`、`logs`、`triggers`、`config`。

需要注意路径与 package 不完全一致：部分 presentation 文件物理位于 `feature/plugin/presentation`，package 仍为 `com.astrbot.android.ui.plugin` 或 `com.astrbot.android.ui.viewmodel`。

## 当前风险与阻塞项

- 本轮未运行 Gradle 或测试，因此不能把本模块状态扩展为构建通过或测试通过。
- `feature/plugin/impl` 当前无生产源码，不应写成 plugin 生产 owner。
- `PluginExecutionHostApi`、compat provider/test harness 和 sample plugin fixture 只作为 compat/test residue 或测试资产理解，不是生产 DI 主线。
- `app/src/test/resources/architecture/app-integration-allowlist.txt` 当前没有 plugin app-integration broad provider allowlist；后续新增 plugin allowlist 必须先证明边界债务合理。
- `global-singleton-allowlist.txt` 当前只保留 QQ stateless mapper；不得为 plugin runtime 增加有状态 singleton allowlist。
- `static-repository-usage-allowlist.txt` 当前没有生产 static repository 例外；plugin 新代码不得回退到 static repository facade。
- ToolSource 当前跨 `feature/plugin/runtime`、`core/runtime-tool`、Resource Center、Config 和 Cron；修改时需要同时看上下游合同，不能只改 provider 私有实现。
- presentation 物理路径与 Kotlin package 不完全一致，新窗口需要路径和 package 双重确认。
