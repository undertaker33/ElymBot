## 模块职责

本模块当前覆盖：

- `:download:api`：下载 task model、owner type、状态、请求/记录 DTO 和 `DownloadManagerPort`。
- `:download:impl`：Room-backed download store、可恢复 HTTP downloader、download queue manager、download foreground service、download notification。
- app-integration 的 download bootstrap binding：把 `DownloadManagerBootstrap` 接入启动 prerequisite。
- app assets runtime 目录：container scripts、Ubuntu rootfs archive、NapCat/QQ/offline installer source assets。
- app `prepareFilteredAssets` 资产过滤规则：控制哪些 runtime 资产进入 APK。
- app `jniLibs/arm64-v8a`：container/proot 脚本运行需要的 native binary 包装。
- container foreground service Android adapter：`ContainerBridgeService`。
- container installer 与 scripts 对资产的消费方式：复制脚本、复制可用 bundled assets、提取 Ubuntu rootfs、启动 NapCat installer fallback。

## 非职责

- `core/runtime-container` 的通用 runtime contract 已在 `core-runtime` 模块确认；本模块只补充 container 资产、脚本、installer 与 foreground service 链路。
- QQ 登录、OneBot 消息链、NapCat 登录状态业务不在本模块收口；后续 `qq-napcat-onebot` 单独治理。
- voiceasset UI、voice clone、TTS voice asset repository 不在本模块收口；这里只确认它们如何使用 `DownloadManagerPort` 和 container scripts。
- plugin package validation、plugin lifecycle、host capability 不在本模块收口；这里只确认 remote plugin package 下载经 `DownloadManagerPort`。
- 不得把下载实现写回 app shell，也不得恢复 `AppDownloadManager` / `AppDownloadManagerBootstrap` 静态 facade。

## 下载主线

当前下载 API：

- `DownloadOwnerType`：`PLUGIN_PACKAGE`、`RUNTIME_ASSET`。
- `DownloadTaskStatus`：`QUEUED`、`RUNNING`、`PAUSED`、`FAILED`、`COMPLETED`、`CANCELED`。
- `DownloadRequest` 要求 `taskKey`、`url`、`targetFilePath`、`displayName`、`ownerId` 非空，并派生 `.part` 临时文件路径。
- `DownloadManagerPort` 暴露 `enqueue`、`observe`、`awaitCompletion`、`resume`、`cancel`。

当前下载实现：

- `RoomDownloadTaskStore` 通过 `DownloadTaskDao` 读写 `DownloadTaskEntity`，映射到 `DownloadTaskRecord`。
- `DefaultDownloadManager` 负责合并已有任务、恢复 queued/running 任务、启动 `DownloadForegroundService`、串行处理 queue、暂停与失败状态回写。
- `UrlConnectionResumableHttpDownloader` 使用 `Range` 与 `If-Range` 继续 `.part` 文件；服务端不支持 range 或校验变化时回退为完整下载。
- `DownloadForegroundService` 是 `@AndroidEntryPoint` service，注入 `DefaultDownloadManager`，在前台通知中处理下载队列。
- `DownloadNotificationController` 使用 notification channel `astrbot_downloads` 和 notification id `2048`。

Manifest 边界：

- `download/impl/src/main/AndroidManifest.xml` 声明 `com.astrbot.android.download.DownloadForegroundService`。
- 该 service `android:exported="false"`，`android:foregroundServiceType="dataSync"`。
- `app/src/main/AndroidManifest.xml` 不直接声明 download impl service；app 只声明通用 foreground service 权限。

启动边界：

- `BootstrapPrerequisitesStartupChain.run()` 调用 `downloadManagerBootstrapPort.ensureReady()`。
- `DownloadManagerBootstrapBinding.kt` 通过 Hilt 把 `DownloadManagerBootstrap` 注入到 `HiltDownloadManagerBootstrapPort`。
- `HiltDownloadManagerBootstrapPort.ensureReady()` 当前是 `Unit`；实际恢复行为仍来自 `DownloadManagerBootstrap` 构造函数中的协程：如果存在 recoverable task，则启动 `DownloadForegroundService`。
- 因此旧记忆中的 constructor side-effect 风险当前仍是代码事实：这是显式风险，不应被文档写成已经完全显式 lifecycle 化。

## Container 资产与 installer 链路

`app/build.gradle.kts` 当前使用 `prepareFilteredAssets` 从 `src/main/assets` 复制到 `build/generated/filtered-assets/main`，并把 main assets source dir 设置为该过滤目录。

当前被排除的 runtime 资产包括：

- `runtime/assets/offline-rootfs-overlay.tar.xz`
- `runtime/assets/NapCat.Shell.zip`
- `runtime/assets/QQ.deb`
- `runtime/assets/launcher.cpp`
- `runtime/assets/offline-debs.tar`
- `runtime/assets/napcat-installer.sh`
- Sherpa / VITS / Kokoro 等大模型资产目录或 ONNX 文件

当前未被该列表排除、会随过滤 assets 保留的关键 container 资产是：

- `runtime/assets/ubuntu-rootfs.tar.xz`
- `runtime/scripts/**`

`ContainerRuntimeInstaller` 当前行为：

- 复制 `container_env.sh`、`bootstrap_container.sh`、`prepare_tts_assets.sh`、`clear_tts_assets.sh`、`convert_tencent_silk.sh`、`encode_tencent_silk.py`、`root_launcher.sh`、`start_napcat.sh`、`logout_qq.sh`、`stop_napcat.sh`、`status_napcat.sh` 到 `filesDir/runtime/scripts`。
- 从 `applicationInfo.nativeLibraryDir` 为 `proot`、`loader`、`busybox`、`libtalloc.so.2`、`bash`、`sudo` 建立 runtime symlink。
- 尝试复制 `ubuntu-rootfs.tar.xz`、`napcat-installer.sh`、`NapCat.Shell.zip`、`QQ.deb`、`launcher.cpp`、`offline-debs.tar`、`offline-rootfs-overlay.tar.xz` 到 `filesDir/runtime/assets`。
- bundled asset 缺失时只记录 `Bundled asset missing or skipped`，不直接失败。
- `ubuntu-rootfs.tar.xz` 是必需资产；`RootfsExtractor.ensureExtracted(...)` 失败会记录并抛出 `Required Ubuntu rootfs extraction failed`。
- `RootfsOverlayExtractor` 当前存在，但本轮源码搜索未发现被当前 installer 调用；不能把 overlay 应用写成当前执行事实。
- 成功准备 rootfs 后通过 `ContainerBridgeStatePort.applyRuntimeDefaults(...)` 写入默认本地 NapCat bridge 配置。

`root_launcher.sh` 当前事实：

- 先尝试 staging bundled NapCat installer assets。
- bundled installer assets 不可用时 fallback 到 upstream `NAPCAT_INSTALLER_URL`。
- 现有 NapCat install 修复路径会尝试 bundled launcher shim；缺失时不硬失败。
- 通过 progress file 写入 `download-installer`、`installer-downloaded`、`installer-cached` 等 label。

`ContainerBridgeRuntimeSupport` 当前负责把 progress file 映射为 UI/通知可读文案，其中保留了 network-install 相关 label。

## Foreground service 与 bridge adapter

当前存在两个前台 service 边界：

- download service：`download/impl/src/main/java/com/astrbot/android/download/DownloadForegroundService.kt`，由 download impl manifest 声明。
- container service：`app/src/main/java/com/astrbot/android/core/runtime/container/ContainerBridgeService.kt`，由 app manifest 声明，是 app 侧 Android service adapter。

`ContainerBridgeService` 当前通过 Hilt 注入：

- `ContainerBridgeStatePort`
- `ContainerRuntimeInstallerPort`
- `ContainerRuntimeController`
- `RuntimeLogger`
- `BridgeHealthChecker`

`AndroidRuntimeBridgeController` 是 app 侧 adapter，负责按 action 启动 `ContainerBridgeService`。

`ContainerRuntimeModule` 当前绑定：

- `DefaultCommandRunner` -> `CommandRunner`
- `ContainerRuntimeInstaller` -> `ContainerRuntimeInstallerPort`
- `DefaultContainerRuntimeController` -> `ContainerRuntimeController`
- `AndroidRuntimeBridgeController` -> `RuntimeBridgeController`

`QqContainerBridgeStatePortAdapter` 把 `feature/qq/data` 的 `NapCatBridgeStateOwner` 映射到 core container state port。这是跨模块 wiring 事实；QQ 业务语义仍留给后续 `qq-napcat-onebot`。

## Runtime asset 与 plugin 调用者

当前下载调用者：

- `feature/voiceasset/data/RuntimeAssetRepository.kt` 注入 `DownloadManagerPort`，用 `DownloadOwnerType.RUNTIME_ASSET` 下载 Sherpa STT 与 Kokoro TTS archive。
- 同一类还注入 `ContainerRuntimeInstallerPort` 与 `CommandRunner`，通过 `prepare_tts_assets.sh` / `clear_tts_assets.sh` 管理 TTS conversion assets。
- `feature/plugin/runtime/PluginInstaller.kt` 注入 `DownloadManagerPort`，用 `DownloadOwnerType.PLUGIN_PACKAGE` 下载 remote plugin package，并把 `DownloadTaskRecord` 映射为 `PluginDownloadProgress`。

这些事实只确认 download port 的使用边界；voiceasset / plugin 的领域行为仍由后续模块单独确认。

## 当前债务与阻塞项

- `DownloadManagerBootstrap` 当前仍通过构造函数 side effect 恢复 recoverable downloads；`ensureReady()` 本身不执行显式逻辑。文档只能把这写成现状和风险，不能写成已完全收口。
- `RootfsOverlayExtractor` 当前存在但未被 installer 调用；旧文档里如果写成 overlay 已经应用，应视为过期或未确认。
- app source assets 下存在 NapCat/QQ/offline 大资产，但 `prepareFilteredAssets` 当前将其中多个 runtime assets 排除出 APK；文档必须区分 source assets 与 packaged assets。
- container service 仍位于 app root，是 Android service adapter；不要把它误写成 `:core:runtime-container` 内的 Android service。
- 本轮未运行 Gradle 或测试，因此不能把本模块状态扩展为构建通过。
