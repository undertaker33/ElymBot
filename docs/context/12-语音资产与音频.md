## 模块职责

本模块当前覆盖：

- `:feature:voiceasset:api`：runtime asset / TTS voice asset / voice clone ports，以及 `RuntimeAsset*`、`TtsVoiceReferenceAsset`、`TtsVoiceReferenceClip`、`ClonedVoiceBinding`、`VoiceAssetProviderType` 等模型。
- `:feature:voiceasset:data`：`RuntimeAssetStateOwner`、`TtsVoiceAssetRepository`、`VoiceCloneService`、Room mapper 和 `VoiceAssetDataModule` Hilt binding。
- `:feature:voiceasset:presentation`：资产管理页、资产详情页、TTS voice asset 导入/克隆/管理 UI、`RuntimeAssetViewModel` 和相关 Compose 组件。
- `:core:runtime-audio`：通用音频 contract、Sherpa ONNX asset service、on-device TTS catalog、TTS prompt/style helper、Silk audio encoder contract 与当前 Tencent Silk encoder 实现。
- App 侧 audio adapter：`SherpaOnnxBridge`、`AndroidSystemTtsBridge`、`TtsVoiceCatalog`、`AudioRuntimeModule` 和 `CompatChatCompletionAudioRuntimePort`。
- Provider runtime 侧语音桥接：provider probe、voice choices、reference audio import、voice binding、TTS asset state 和 synthesized speech 通过 `ProviderRuntimePort` 暴露给 provider / voiceasset UI。
- Room TTS asset 数据真源与 backup TTS asset snapshot / restore。
- container runtime scripts 中 TTS conversion assets 的 prepare / clear / Tencent Silk conversion 链路。

## 非职责

- `core-runtime` 的 generic runtime 分层已在第 4 模块确认；本模块只补充 audio/voiceasset 现状，不把所有 runtime 重新收口。
- download manager 和 container installer 资产复制主线已在第 5 模块确认；本模块只记录 voiceasset 如何使用 `DownloadManagerPort` 与 TTS 脚本。
- Provider / Config / Bot / Persona 业务仓储和 selected state 已在第 6 模块确认；本模块只记录 provider runtime 如何暴露 STT/TTS 与 voice asset 能力。
- Chat / QQ 消息链分别已在第 7、8 模块确认；本模块只记录它们如何消费 STT/TTS / Silk 能力。
- Backup 主线已在第 11 模块确认；本模块只确认 `ttsAssets` 的数据 owner、snapshot / restore 和 participant 状态。
- 不得把 voiceasset data/presentation 实现写回 app shell、root `data/*`、root `runtime/*` 或旧 static repository。

构建依赖事实：

- `:feature:voiceasset:presentation` 启用 Compose + Hilt + KSP，依赖 `:core:ui`、`:feature:provider:api` 和 voiceasset API；不得直接 import DAO、data repository 静态入口或 provider runtime implementation。
- `:core:runtime-audio` 启用 KSP + Hilt，依赖 `:core:common`、`:core:logging`、`:core:runtime-container`、commons-compress、coroutines android 和 `javax.inject`。
- root `build.gradle.kts` 当前有 `moduleVoiceAssetBuild` / `moduleVoiceAssetCheck`，覆盖 voiceasset 三个 feature 模块。
- `app-integration/build.gradle.kts` 以 `api(project(":core:runtime-audio"))` 与 `api(project(":feature:voiceasset:api"))` 接入 contract，以 `implementation(project(":feature:voiceasset:data"))` 接入 data wiring。
- `app/build.gradle.kts` 生产直接依赖 `:feature:voiceasset:presentation`；app 侧保留 Android audio bridge 和 Hilt module，不直接成为 voiceasset data owner。

`RuntimeAudioBoundaryContractTest` 当前还要求 `:app` 不直接依赖 `:core:runtime-audio`，runtime audio 由 app-integration 消费，同时 app 侧 Android adapter 仍位于 `:app`。

## VoiceAsset API 与 Data 真源

当前 voiceasset API 暴露三个生产 port：

- `RuntimeAssetPort`：runtime asset 状态、TTS conversion assets、on-device framework / STT / TTS model 下载与清理。
- `TtsVoiceAssetPort`：TTS reference audio 列表、导入、provider binding、重命名、清理、删除、snapshot / restore。
- `VoiceCloneRuntimePort`：面向 provider profile 与 reference asset 的 voice clone 执行入口。

`VoiceAssetDataModule` 通过 Hilt 绑定：

- `RuntimeAssetStateOwner -> RuntimeAssetPort`
- `TtsVoiceAssetRepository -> TtsVoiceAssetPort`
- `VoiceCloneService -> VoiceCloneRuntimePort`

当前数据真源与持久化事实：

- `TtsVoiceAssetRepository` 是 `@Singleton` injectable class，不再是 `object`，不通过 `AstrBotDatabase.get(context)` 自举，也不保留 `graphInstance` 静态转发。
- TTS reference audio 导入复制到 `filesDir/assets/tts-reference-audio`，支持 WAV、MP3、M4A、AAC，并读取 duration / sample rate。
- 多 clip asset 以 `TtsVoiceReferenceClip` 保存；provider binding 以 `ClonedVoiceBinding` 保存。
- Room owner 位于 `core/db`：`tts_voice_assets`、`tts_voice_clips`、`tts_voice_provider_bindings` 与 `TtsVoiceAssetAggregateDao`。
- `TtsVoiceAssetMappers.kt` 在 feature data 中负责 Room aggregate 与 API model 的转换。
- `TtsVoiceAssetRepository` 仍会在新表为空时从 legacy `SharedPreferences` 的 `tts_voice_assets/assets_json` seed，但当前 production 操作走 Room aggregate。

## Runtime Asset 与 on-device audio

`RuntimeAssetStateOwner` 当前管理五类 asset：

- `tts`：TTS conversion assets，通过 container 脚本 `PREPARE_TTS_ASSETS` / `CLEAR_TTS_ASSETS` 管理，marker 为 rootfs 内 `/root/.astrbot-tts-assets-ready`。
- `on-device-framework`：Sherpa ONNX framework marker，`DefaultSherpaOnnxAssetService` 写入 `runtime/sherpa-onnx/.framework-ready`。
- `on-device-stt`：Sherpa ONNX Paraformer STT archive，下载 task key 为 `asset:stt:paraformer-zh-small-2024-03-09`。
- `on-device-tts`：Kokoro local TTS archive，下载 task key 为 `asset:tts:kokoro-int8-multi-lang-v1_1`。
- `tts-voice-assets`：云端 TTS voice asset 管理入口，列表项本身禁用 download / clear action，具体导入与克隆走 voice asset UI。

`DefaultSherpaOnnxAssetService` 是当前 Sherpa asset production service；旧 `SherpaOnnxAssetManager object` 已由合同测试要求退休。它负责：

- framework / STT / Kokoro 状态判断。
- framework 激活与清理。
- STT / Kokoro archive 下载或从已下载 archive 安装。
- tar.bz2 解压到 staging 后替换 target，并检查 archive entry 不逃逸目标目录。
- 清理旧 TTS 目录：`matcha`、`melo`、`fanchen-c`、`zh-ll`。

`OnDeviceTtsCatalog` 当前只声明 `kokoro` 模型，并提供内置中文 speaker voice 列表。

## App audio adapter 与 LLM/Provider 桥接

App 侧仍保留 Android-specific adapter：

- `SherpaOnnxBridge`：使用 `SherpaOnnxAssetService`，负责 on-device STT PCM 解码、Sherpa recognizer cache、Kokoro TTS 初始化与 WAV attachment 生成。
- `AndroidSystemTtsBridge`：使用 Android `TextToSpeech` 生成 WAV attachment，是 injectable instance，不是 global object。
- `TtsVoiceCatalog`：为 OpenAI / Bailian / MiniMax / Sherpa ONNX TTS provider 生成可选 voice choices。
- `AudioRuntimeModule`：提供 `SherpaOnnxAssetService`、`SherpaOnnxBridge` 和 `AudioRuntimePort`。
- `CompatChatCompletionAudioRuntimePort`：当前仍把云端 STT/TTS 委托给 `ChatCompletionService`，同时通过 `SherpaOnnxBridge` 暴露 Sherpa readiness；这是显式 compat adapter，不是第二套 DI。

`HiltLlmProviderProbePort` 和 `LlmMediaService` 均通过 injected `AudioRuntimePort` 进行 STT/TTS，不应在 feature 或 UI 中直接调用 `ChatCompletionService` 的媒体方法。

Provider runtime 当前通过 `DefaultProviderRuntimePort` 暴露：

- `probeSttSupport(...)`
- `probeTtsSupport(...)`
- `listVoiceChoicesFor(...)`
- `voiceAssets`
- `importReferenceAudio(...)`
- `saveVoiceBinding(...)`
- `renameVoiceBinding(...)`
- `clearReferenceAudio(...)`
- `deleteReferenceClip(...)`
- `deleteVoiceBinding(...)`
- `ttsAssetState(...)`
- `isSherpaFrameworkReady()` / `isSherpaSttReady()`
- `synthesizeSpeech(...)`

Voiceasset presentation 只依赖 `feature/provider/api` contract 和 voiceasset API，不依赖 `feature/provider/runtime` implementation。

## Voice Clone 与 provider 支持

`VoiceCloneService` 当前支持：

- `BAILIAN_TTS`：要求 provider model 以 `qwen3-tts-vc` 开头，请求 `/services/audio/tts/customization`，返回 `voice_id` / `voice` / `id`。
- `MINIMAX_TTS`：先上传 reference audio 到 `/files/upload`，再调用 `/voice_clone`，并处理 `base_resp.status_code`。

当前 clone 输入来自 `TtsVoiceReferenceAsset`，会优先合并多个本地 WAV clip；如果 clip 不满足合并条件，则使用最长本地 clip、legacy `localPath` 或 `remoteUrl`。其他 provider type 会抛出不支持异常，不得写成通用 provider clone 已覆盖。

## UI 与导航入口

当前 UI 入口：

- `AssetManagementScreen`：runtime asset 列表。
- `AssetDetailScreen`：runtime asset 详情；当 asset 为 `tts-voice-assets` 时切入 voice asset 管理。
- `RuntimeAssetViewModel`：只注入 `RuntimeAssetPort`，管理 runtime asset download / clear。
- `AssetScreenBindingsViewModel`：注入 `ProviderRuntimePort` 与 `VoiceCloneRuntimePort`，把 provider options、voice assets、import、binding、clone 操作暴露给 Compose。
- `ui/voiceasset/*`：overview、import、clone、reference library、cloned voices、face switcher 等组件。

导航入口由 app navigation 接入：

- `asset-management`
- `asset-management/{assetId}`
- settings / Me 页面中的 Assets 入口
- backup UI 中的 `backup/tts` 属于第 11 模块确认过的 settings backup route，本模块只确认 TTS asset data owner。

需要注意路径与 package 不完全一致：voiceasset presentation 的物理路径在 `feature/voiceasset/presentation`，但 package 仍包含 `com.astrbot.android.ui.settings` 与 `com.astrbot.android.ui.voiceasset`。

## QQ / Chat / Silk 消费边界

Chat runtime 和 QQ runtime 消费音频能力，但不是 voiceasset data owner：

- App Chat 的 prepared reply 通过 runtime binding 使用 TTS provider、`ttsVoiceId` 和 `ttsReadBracketedContent`。
- QQ runtime 的 `QqStreamingReplyService` 根据 config 生成单段或多段 voice reply attachment。
- `QqRuntimeServicesModule` 注入 `SilkAudioEncoder`，并把 `silkAudioEncoder::encode` 放入 `QqOneBotRuntimeDependencies`。
- `SilkAudioEncoder.kt` 当前位于 `core/runtime-audio`，其中 `TencentSilkEncoder` 通过 container script `CONVERT_TENCENT_SILK` 转换 QQ TTS 音频；脚本发现 TTS conversion assets 未准备时会提示到 Asset Management 下载。

不要把 QQ 发送协议、OneBot payload、NapCat 登录或 Chat conversation 语义写成本模块 owner。

## Backup 覆盖面

当前 TTS voice asset backup 事实：

- `AppBackupModuleKind.TTS_ASSETS`、manifest `ttsAssets` 和 `BackupParticipantRegistry` 中的 `tts-assets` 已存在。
- `AppBackupRepository` 通过 injected `TtsVoiceAssetPort` 调用 `snapshotAssets()` 和 `restoreAssets(...)`。
- 导出 manifest 中 `ttsAssets` 记录 count、hasFiles、files、records。
- restore stage 中 `ttsAssets` 有 apply / rollback，失败时可恢复到 current snapshot。
- `ModuleBackupSupportTest`、`TtsBackupArchiveTest`、`AppBackupJsonTest`、`AppBackupManifestAdaptersTest` 与 `AppBackupImportPlannerTest` 覆盖 TTS asset 相关导入预览和 manifest / archive 行为。

此处只代表 TTS voice asset 覆盖；Resource Center、Cron 和 Plugin backup gaps 仍按第 11 模块记录，不得因为 `tts-assets` supported 就写成全部 runtime asset 或全部资源已支持 backup。
