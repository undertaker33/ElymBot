## 4. Application / AppBootstrapper / Hilt

当前启动入口是：

1. `AstrBotApplication`
2. Hilt 注入 `AppBootstrapper`
3. `AppBootstrapper.bootstrap()`
4. `AppStartupRunner.run()`
5. `di/startup/*` 中的固定 startup chains 按顺序执行

旧入口状态：

- `AstrBotAppContainer.kt` 已不再是当前生产入口
- `ElymBotAppContainer.kt` 已被 `3bf13d5` 重命名/收敛为 `AppBootstrapper.kt`
- `ComposeViewModelHelpers.kt` 已删除
- UI 侧不再通过 `astrBotViewModel(...)` / `viewModelFactory` 取依赖，改用 Hilt

生产依赖注入限制：

- 生产依赖注入唯一合法方法是 Hilt。合法入口是 `@HiltAndroidApp`、`@AndroidEntryPoint`、`@HiltViewModel`、`@HiltWorker`、constructor injection、`di/hilt/*` module，以及由 Hilt graph 注入/提供的 production port。
- backup production ports 在 `app-integration/src/main/java/com/astrbot/android/di/BackupDataPortAdapter.kt` + `di/hilt/BackupModule.kt`；`di/RuntimeContextDataPorts.kt`、`di/ContainerBridgeStatePorts.kt` 是对应 runtime/container production port 实现。它们都不是第二套 DI 框架，新增生产依赖不要再通过 registry/service locator 安装。
- `AppBootstrapper` 现在只承担启动薄壳；不要把它扩回手写 app container、dependency holder 或 ViewModel factory。
- `AppStartupChainContractTest` 明确禁止把 startup 改成 `Set<AppStartupChain>` / `@IntoSet` 这种无序发现模型。
- `StrictHiltOnlyContractTest`、`StrictHiltOnlyFinalContractTest`、`StrictHiltOnlyStartupHotspotSourceTest` 会继续保护这条 Hilt-only 主线。
- 测试 override 只能留在 `src/test` / `src/androidTest` 或明确 test hook 中，不要把测试兼容 contract 写成生产依赖入口。

容器/runtime 入口新增变化：

- `ContainerBridgeService` 现在是 `@AndroidEntryPoint`
- `core/runtime/container/ContainerRuntimeEntryPoint.kt` 是 controller/service 获取 `ContainerBridgeStatePort` 与 `ContainerRuntimeInstaller` 的 Hilt entry point
- `RuntimeLaunchStartupChain` 会在 app 启动时异步 warm up `ContainerRuntimeInstaller`

注意：`AppBootstrapper` 仍是启动 composition root，但它已经不再手写 assemble repository init / guard 注册 / QQ runtime 启动。新增生产依赖优先走 Hilt module、startup chain 或显式 production port，不要再新增手写 app container 字段。

## 10. 易错点

- 新增页面优先落到 `feature/<module>/presentation`，不要落回 root `ui`。
- 新增跨 feature 启动初始化时，先考虑 startup chain、Hilt module 或已有 production port，不要直接在 `core` 里 import feature，也不要把逻辑塞回 `AppBootstrapper`。
- 新增生产 DI 依赖时只能走 Hilt DI：优先看 `di/hilt/*`、constructor injection、`@HiltViewModel` / `@HiltWorker` 和已有 production port；不要恢复 `AstrBotAppContainer` / `ElymBotAppContainer`、`ViewModelProvider.Factory`、registry/service locator 或 static `install*` / `configure*` 回调。
- `:app-integration` 是 wiring 边界，不是新的业务垃圾桶；不要把 feature Screen/ViewModel、runtime loop、payload parser 或 repository mutation workflow 塞进去。
- QQ 生产启动入口是注入的 `QqBridgeRuntime.start()`；`QqOneBotBridgeServer.kt` 仍是 QQ bridge 关键文件，但 `QqOneBotBridgeServer` object 主要保留 test/compat 静态入口。
- 不要在生产路径恢复 `QqOneBotBridgeServer.installRuntimeDependencies(...)` 或 `ScheduledTaskRuntimeExecutor.configureRuntimeDependenciesProvider(...)`。
- `ContainerBridgeService` / `ContainerBridgeController` 当前通过 `ContainerRuntimeEntryPoint` 取 Hilt-owned runtime 依赖；不要在 feature 层自己拼 container installer/state。
- 改 Manifest、network security、WorkManager 初始化元数据或 deep link 行为时，要同步 `AndroidManifestRuntimeContractTest` 与 `MainActivityPluginDeepLinkTest`。

## 11. Phase 24-26 App Shell / App Integration 终态口径

- `:app` 当前只按 Android 壳层理解：Manifest、Application、Activity、navigation root、theme/resources、app chrome 与 thin platform adapter。
- `:app` 不再直接依赖 feature data/runtime/impl 生产主线；已拆出的 QQ data/runtime/presentation、voiceasset data/presentation、provider runtime、backup wiring 都不应回填成 app direct dependency。
- `:feature:qq:data`、`:feature:qq:runtime`、`:feature:qq:presentation` 已承载 QQ production 主线；`:feature:qq:impl` 不再是当前生产 owner。
- `:feature:voiceasset:data` 与 `:feature:voiceasset:presentation` 已承载资产生产主线；voiceasset presentation 不依赖 provider runtime implementation。
- `:app-integration` 仍是 Hilt wiring 边界：允许承载 `Module` / `Binding` / `Adapter` / `Contributor` 形态文件，不依赖任何 feature presentation，不引入 Compose，不承载 `ProcessBuilder`、long-running loop 或业务状态机。
- `architectureCheck` 当前包含 `:architecture-tests:test`；Gradle convention 相关变更还要补跑 `:build-logic:check`。
