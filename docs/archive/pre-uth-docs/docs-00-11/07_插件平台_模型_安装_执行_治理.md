## 3. V1 / V2 当前边界

### 3.1 V1 legacy freeze

V1 插件 dispatch 被显式冻结在：

- `feature/plugin/runtime/PluginV1DispatchAdapter.kt`

`PluginRuntimeFacade` 通过这个 adapter 暴露 frozen V1 boundary。`PluginV1LegacyAdapter.kt` 已删除，新 runtime 代码不应继续隐式扩展 V1 路径。

`d060c5e` 下触发器口径继续收紧：

- `PluginTriggerContracts.onlineHostTriggers`：`OnPluginEntryClick`、`OnCommand`、`BeforeSendMessage`、`AfterModelResponse`
- `PluginTriggerContracts.residualCompatOnlyTriggers`：`OnMessageReceived`、`OnSchedule`、`OnConversationEnter`
- `ExternalPluginTriggerPolicy` 已不再作为 App Chat / QQ 在线主链的 gating 入口
- `PluginLegacyTriggerRetirementContractTest` 防止 residual V1 triggers 回流到生产主线

### 3.2 V2 仍是主运行时

V2 主文件仍在 `feature/plugin/runtime/PluginV2*`：

- lifecycle
- dispatch engine
- message contracts
- LLM pipeline coordinator
- tool registry
- tool loop coordinator
- QuickJS session/runtime loader

共享 LLM orchestration 当前路径：

- `feature/plugin/runtime/RuntimeLlmOrchestratorPort.kt`
- `feature/plugin/runtime/DefaultRuntimeLlmOrchestrator.kt`
- `feature/plugin/runtime/RuntimeOrchestrator.kt`

`DefaultRuntimeLlmOrchestrator` 是真实实现；`RuntimeOrchestrator` 只是兼容壳。App Chat / QQ / scheduled task 都应通过 `RuntimeLlmOrchestratorPort` 注入到共享 LLM pipeline，不要在新代码里直接依赖静态 `RuntimeOrchestrator`。

Hilt 生产装配在：

- `di/hilt/RuntimeServicesModule.kt`
  - `provideRuntimeLlmOrchestratorPort(): RuntimeLlmOrchestratorPort = DefaultRuntimeLlmOrchestrator()`
  - `providePluginExecutionEngine()`
  - `providePluginHostCapabilityGateway()`
  - `provideAppChatPluginRuntime(): AppChatPluginRuntime = EngineBackedAppChatPluginRuntime(...)`
  - `provideScheduledTaskRuntimeDependencies(...)`
  - `provideQqOneBotRuntimeDependencies(...)`
  - `provideQqBridgeRuntime(...)`
  - `provideQqScheduledMessageSender(...)`
- `di/hilt/PluginRuntimeModule.kt`
  - 提供 `PluginConfigStorage`
  - 提供 `PluginHostConfigResolver`
  - 提供 `PluginStateStore`
  - 提供 `PluginRuntimeArtifactCleaner`
  - 提供 `PluginDataCleanupService`
  - 提供 `PluginRuntimeLogBus`
  - 提供 `PluginFailureStateStore` / `PluginScopedFailureStateStore` / `PluginScheduleStateStore`
  - 提供 `PluginV2ActiveRuntimeStore`
  - 提供 `PluginV2LifecycleManager`
  - 提供 `PluginV2DispatchEngine`
  - 提供 `PluginV2RuntimeLoader`
- `di/hilt/PluginHostCapabilityModule.kt`
  - 绑定 `PluginExecutionHostResolver`
  - 提供 `DefaultPluginExecutionHostOperations`
  - 提供 `PluginHostCapabilityGatewayFactory`

当前要把这三层分清：

- Hilt-owned production mainline：`PluginRuntimeModule` + `RuntimeServicesModule` + `PluginHostCapabilityModule`
- Hilt-to-compat residue：`PluginRuntime*Provider.installFromHilt(...)` 这一类 install residue
- compat static residue：`PluginExecutionHostApi` / `createCompatPluginHostCapabilityGatewayFactory()`

`32dd105` 下 `PluginRuntimeDependencyBridge.kt` 已删除。当前不要再把任何 compat residue 写成第二套生产 DI 框架。

插件入口点击执行也继续收口到：

- `feature/plugin/runtime/PluginEntryExecutionService.kt`
- `feature/plugin/runtime/ExternalPluginRuntimeCatalog.kt`

新代码不要再直接在 presentation/runtime 层自己扫 active runtime store 决定 entry click 执行。

### 3.3 插件配置、状态与卸载清理

`d060c5e` 后插件配置/状态从 repository 内部逻辑继续拆出：

- `feature/plugin/data/config/PluginConfigStorage.kt`
  - 负责读取/保存 `plugin_config_snapshots`
  - 负责解析已安装插件的 static config schema / settings schema 路径
- `feature/plugin/data/config/PluginHostConfigResolver.kt`
  - 合并 static defaults、settings schema defaults、core config、extension config
  - 向 host API 提供 `mergedSettings`
- `feature/plugin/data/state/PluginStateStore.kt`
  - 生产实现是 `RoomPluginStateStore`
  - 存储落到 `plugin_state_entries`
  - scope 分为 `plugin` 与 `session`
- `feature/plugin/data/state/PluginStateValueCodec.kt`
  - 只接受 JSON 可序列化值，拒绝非有限数字等非法值
- `feature/plugin/domain/cleanup/PluginDataCleanupService.kt`
  - 卸载时始终清理 runtime artifact 与安装包/解压目录
  - 只有 `PluginUninstallPolicy.REMOVE_DATA` 才删除 config snapshot、state entries 和 private workspace

不要再把 `FeaturePluginRepository` 写成插件配置/状态/卸载文件清理的唯一真源；它现在更多是 install/catalog 状态 owner，并把配置、状态、清理委托给 Hilt-owned services。

### 3.4 Bootstrap host storage API

`PluginV2BootstrapHostApi` 当前向 QuickJS bootstrap 暴露：

- `getSettings` / `getPluginSettings` / `readSettings` / `getConfig`
- `storage.plugin.get/set/remove/keys/clear`
- `storage.session.get/set/remove/keys/clear`

`storage.session` 依赖消息事件里的 `sessionUnifiedOrigin`。App Chat 与 QQ command / message ingress 已把 `MessageSessionRef.unifiedOrigin` 放入 plugin event extras；没有当前会话上下文时会返回 structured error `missing_session_scope`。

状态 key 会 trim，不能为空，长度不能超过 128；value 会通过 `PluginStateValueCodec` 编码成 JSON，Room 侧单值大小限制为 64 KB。不要把 plugin storage 写成任意对象或无限大小 KV。

## 4. ToolSource 新边界

ToolSource 当前路径：

- `feature/plugin/runtime/toolsource/*`

同时新增 core contract：

- `core/runtime/tool/ToolSourceContracts.kt`
- `core/runtime/tool/ToolDescriptorCachePolicy.kt`

`fb8e7ff` 下 `FutureToolSourceRegistry` 已变成 Hilt `@Singleton` + context resolver 主线：

- `FutureToolSourceContextResolver` 根据 `ConfigRepositoryPort + ResourceCenterPort` 生成 `ToolSourceContext`
- `collectContractDescriptors(...)` 返回 core `ToolDescriptor`
- `collectToolDescriptors(...)` 是 legacy plugin descriptor 兼容路径
- `PluginToolSourceBridge` 继续负责 descriptor cache
- Resource Center / tool skill 的 JSON schema 通过 `core/runtime/tool/ToolJsonValueNormalizer.kt` 统一规整，避免多处手写 JSONObject / JSONArray 递归转换

默认 providers：

- `McpToolSourceProvider`
- `SkillToolSourceProvider`
- `ActiveCapabilityToolSourceProvider`
- `ContextStrategyToolSourceProvider`
- `WebSearchToolSourceProvider`

## 5. MCP 当前边界

`McpToolSourceProvider` 当前只支持：

- `streamable_http`

明确不支持：

- legacy SSE 作为工具源可用路径
- stdio subprocess MCP

MCP 资源来自 `ToolSourceContext.mcpServers`，而这个 context 由 Resource Center projections 投影生成。

## 6. Skill 当前边界

Skill 已拆成两类：

- Prompt Skill
  - 来自 `PromptSkillProjection`
  - 进入 `PromptAssembler`
- Tool Skill
  - 来自 `ToolSkillProjection`
  - 进入 `SkillToolSourceProvider`
  - `resultTemplate` 可用 `{{key}}` 替换 payload 字段

不要再把所有 `SkillEntry` 都理解成“prompt 注入”。

## 7. Active Capability 当前边界

Active capability 仍暴露：

- `create_future_task`
- `delete_future_task`
- `list_future_tasks`
- `pause_future_task`
- `resume_future_task`
- `list_future_task_runs`
- `update_future_task`
- `run_future_task_now`

但创建未来任务的实现已抽到：

- `ActiveCapabilityRuntimeFacade`

它负责：

- 校验 note
- 解析 run_at / cron_expression
- 解析 target context
- 创建 `CronJob`
- 调 scheduler
- pause / resume / update / run-now 管理
- list recent execution records
- 返回 structured JSON 或 structured error

`fb8e7ff` / `e495263` 下还要按这条装配路径理解：

- `ActiveCapabilityRuntimeFacade` 通过 `CronJobRepositoryPort + CronSchedulerPort` 注入
- `run_future_task_now` 通过 `CronJobRunNowPort -> CoordinatorCronJobRunNowPort -> CronJobRunCoordinator.runDueJob(trigger = "run_now")`
- `ActiveCapabilityToolSourceProvider` 是 `@Inject` provider，不再走 `initialize(application)` 或 appContext 全局回填
- target context 仍是必填语义，缺失会返回 structured error `missing_context`
- 当前 ingress trigger 是 `SCHEDULED_TASK` 时，active capability tools 会隐藏，避免 scheduled task 执行时再次创建/管理任务

缺失 target context 会返回 `missing_context`，不是静默创建。

`e495263` 还新增 host fallback 边界：

- `ScheduledTaskIntentGuard`：在 proactive enabled 且文本明显是提醒/定时意图时，可用自然语言 parser 创建 run-once 或 recurring task
- `ScheduledTaskIntentFallbackResponder`：如果插件 LLM pipeline 没有执行 `create_future_task`，会尝试让 host guard 兜底创建，并通过 provider follow-up 生成确认回复
- 这条 fallback 是 App Chat / QQ 平台回调里的 host 行为，不是插件 V1 trigger，也不是绕过 ToolSource 的第二套常规调度入口

## 11. Web Search

Web search 仍在：

- `feature/plugin/runtime/toolsource/WebSearchToolSourceProvider.kt`

但搜索策略与 provider 真源已迁到 core：

- `core/runtime/search/UnifiedSearchContracts.kt`
- `core/runtime/search/UnifiedSearchCoordinator.kt`
- `core/runtime/search/WebSearchTriggerRules.kt`
- `core/runtime/search/WebSearchPromptGuidance.kt`
- `core/runtime/search/api/*`
- `core/runtime/search/local/*`
- `feature/provider/runtime/search/ProviderRepositorySearchProvider.kt`
- `di/hilt/SearchRuntimeModule.kt`

当前执行顺序：

1. `WebSearchToolSourceProvider` 校验 config 的 `webSearchEnabled`
2. 构造 `UnifiedSearchRequest`
3. 通过 `UnifiedSearchPort` 先尝试 configured search providers
4. 如果没有可用 configured provider 或结果不可接受，再走 `LocalMetaSearchFallbackProvider`
5. 返回结构化结果和 diagnostics

`a6ffa95` 曾引入 `core/runtime/search/html/*`；`b790abce` 已删除该 HTML fallback，当前不要再把 `HtmlFallbackSearchProvider` 写成主线。当前 local fallback 是：

- engines：Sogou、Bing、Baidu Web Lite、DuckDuckGo Lite、Bing News
- crawler：`DefaultContentCrawlerLite`
- safety/prune/markdown：`UrlSafetyPolicy`、`QueryAwareContentPruner`、`MarkdownLiteGenerator`
- general 查询允许低相关 last resort；news/weather 仍要求相关结果，不接受低相关 fallback

`e495263` 下 Web Search 的生产装配也要按 Hilt 主线理解：

- `WebSearchToolSourceProvider` 通过 `UnifiedSearchPort + FutureToolSourceContextResolver` 注入
- `UnifiedSearchPort` 来自 `SearchRuntimeModule`
- `RuntimeNetworkTransport` 来自 `di/hilt/RuntimeNetworkModule.kt`
- 不要把 web search 写回 plugin 私有 search policy、全局 transport registry 或静态 config self-resolve

## 13. 易错点

- 新插件模型文件在 `feature/plugin/model`，但 package 可能仍是 `com.elymbot.android.model.plugin`。
- V1 是 freeze boundary，不要继续往 V1 路径加新能力。
- residual V1 triggers 只能留在 compat-only 边界；不要把 `OnMessageReceived`、`OnSchedule`、`OnConversationEnter` 写回 App Chat / QQ 在线生产主线。
- ToolSource 要同时考虑 core contract 和 plugin descriptor 兼容路径。
- MCP 只写 `streamable_http` 现状，不要把 SSE/stdio 写成可用。
- 新 LLM runtime wiring 走 `RuntimeLlmOrchestratorPort` / `DefaultRuntimeLlmOrchestrator`；`RuntimeOrchestrator.kt` 只当兼容壳。
- 插件 installer/catalog/subscription/install intent 的 production wiring 在 `PluginProvisioningModule.kt`，不是旧 dependency helper。
- `PluginRuntimeDependencyBridge.kt` 已删除；`PluginRuntime*Provider.installFromHilt(...)` 和 `createCompatPluginHostCapabilityGatewayFactory()` 都只是 compat/test residue，不要写成 production mainline。
- `PluginHostCapabilityGateway` 当前应通过 `PluginExecutionHostResolver` / `PluginHostCapabilityGatewayFactory` 装配；不要回到 direct `PluginExecutionHostApi.*`。
- `PluginRuntimeFacade` 现在只是很薄的 `PluginRuntimePort` 壳，不再暴露 V1 boundary；不要把它误当成新的 plugin 运行时主入口。
- 插件配置/状态的当前生产路径是 `PluginConfigStorage` / `PluginHostConfigResolver` / `PluginStateStore`，不是 `FeaturePluginRepository` 静态入口。
- `storage.session` 必须有 `sessionUnifiedOrigin`；缺上下文时应按 `missing_session_scope` 处理，不要 silent fallback 到 plugin scope。
- active capability tools 在 scheduled task trigger 下会隐藏；不要让 scheduled task 执行时再暴露 `create_future_task` 形成自我递归。
- `feature/plugin/runtime/toolsource/search/*` 已删除；Web Search 现状在 `core/runtime/search/*`。
