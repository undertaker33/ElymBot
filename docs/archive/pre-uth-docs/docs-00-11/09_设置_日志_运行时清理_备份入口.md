## 5. Cron Jobs 入口

Cron UI 已迁到：

- `feature/cron/presentation/CronJobsScreen.kt`
- `feature/cron/presentation/CronJobsViewModel.kt`
- `feature/cron/presentation/CronJobEditorDraft.kt`
- `feature/cron/presentation/CronJobsPresentationController.kt`

Cron ViewModel 不应直接操作 runtime scheduler 或旧 repository singleton。`fb8e7ff` 下当前主线是：

- `CronJobsViewModel` 通过 Hilt 注入 `CronJobRepositoryPort`
- 同时注入 `BotRepositoryPort`、`ConversationRepositoryPort`、`ConfigRepositoryPort`、`ProviderRepositoryPort`
- 创建任务时先解析默认 target context，再交给 `CronJobsPresentationController`
- `ActiveCapabilityTargetContext` 仍是必填语义，缺失上下文不会静默创建 future task

`113dc12` / `e495263` 后 Cron UI 和管理能力还要按新口径理解：

- 创建/编辑 UI 现在显式处理 `runAt`、`runOnce`、`platform`、`conversationId`
- job card 支持 pause/resume 和查看 runs；run history 来自 `cron_job_execution_records`
- `CronJobsPresentationController` 暴露 `pauseJob(...)`、`resumeJob(...)`、`listRuns(...)`
- `CronJobUseCases` 还提供 `runJobNow(...)`，但当前主要由 active capability 的 `run_future_task_now` 工具调用

## 7. Backup 入口

Backup UI 已迁到：

- `feature/settings/presentation/BackupScreen.kt`

Backup core：

- `core/db/backup/*`

当前 backup 入口和恢复语义要按 port closure 后口径理解：

- `app-integration/src/main/java/com/elymbot/android/di/BackupDataPortAdapter.kt` 与 `di/hilt/BackupModule.kt` 是生产 backup wiring 真源
- `AppBackupService` / `ConversationBackupService` 是 Hilt 注入入口，内部私有持有实例级 repository；生产不再通过静态 `AppBackupRepository` / `ConversationBackupRepository` object 调度
- `AppBackupRepository.restore(...)` 会把 `selectedBotId` / `selectedConfigId` 一起纳入恢复与 rollback
- conversation restore 已切到 durable 路径：`restoreSessionsDurable(...)`
- static repository usage allowlist 当前为 0；如果看到 backup compat repository，先确认是不是 test/compat seam，不要把它写回 production mainline。

当前备份仍未覆盖：

- Resource Center
- Cron jobs
- Cron execution records
- Config 的 context/MCP/skills 历史兼容字段

所以不要在 UI 文案或回归说明里把“完整备份”理解成所有新表都可恢复。

`e495263` 后 backup 已覆盖 Config 的 `includeScheduledTaskConversationContext`，但仍不覆盖 Resource Center、Cron 和 plugin config/state。

## 9. 易错点

- Resource Center 是新入口，不是旧 Config 页面的一段表单。
- `ui/settings/CronJobsScreen.kt` 已删除；当前物理路径在 `feature/cron/presentation`。
- `ui/settings/MeScreen.kt` 已删除；当前物理路径在 `feature/settings/presentation`。
- Settings shell 不应 import feature presentation 或 runtime business 对象。
- Cron Worker 当前是 Hilt worker，不要手写 WorkManager factory 或绕过 `HiltWorkerFactory`。
- `CronJobsViewModel` 当前是 injected port + target context 主线，不要再把它写回“直接抓 `CronJobScheduler` / 全局 repository 单例”的旧口径。
- active capability 管理 Cron 时必须通过 `CronJobRepositoryPort` / `CronSchedulerPort` / `CronJobRunNowPort`，不要在 UI 或 ToolSource 里 direct new scheduler/coordinator。
- Backup 入口虽然支持 selected bot/config + conversation durable rollback，但仍不覆盖 Resource/Cron 新表；不要把“恢复更稳”误写成“覆盖更全”。
- 不要把 `AppBackupDataRegistry`、`ConversationBackupDataRegistry`、`ContainerBridgeStateRegistry` 写成当前 settings/runtime 入口；当前 backup production ports 在 `app-integration`，runtime/container ports 在对应 `di/*Ports.kt`。
- `ContainerBridgeService` / `ContainerRuntimeInstaller` 当前是 runtime 入口的一部分；不要把 NapCat/container install 口径写回“手动点击后才第一次准备脚本”的旧认知。
- 日志与运行时清理现在既要看 `RuntimeLogCleanupRepository`，也要看 `RuntimeLogRepository` 本身的持久化与测试口径。
