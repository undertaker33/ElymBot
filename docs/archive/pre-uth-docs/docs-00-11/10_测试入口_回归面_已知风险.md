## 3. 当前新增高风险回归面

### 3.1 feature-first 边界

新增/迁移文件时最容易违反：

- core import feature
- feature domain import Android/Compose/legacy singleton
- feature presentation 直接 import old data/runtime
- root ui/data/runtime 又长出业务代码

这类问题优先由 architecture tests 兜底。

### 3.2 Hilt-only 唯一 DI 方法

`3bf13d5 ~ e495263` 生产代码已迁到：

- `@HiltAndroidApp`
- `@AndroidEntryPoint`
- `@HiltViewModel`
- `@HiltWorker`
- `di/AppBootstrapper.kt`
- `di/startup/*`
- `di/hilt/*`
- `app-integration/src/main/java/com/elymbot/android/di/BackupDataPortAdapter.kt`
- `app-integration/src/main/java/com/elymbot/android/di/hilt/BackupModule.kt`
- `di/RuntimeContextDataPorts.kt`
- `di/ContainerBridgeStatePorts.kt`

当前依赖注入限制已经升级成硬约束：生产依赖注入唯一合法方法是 Hilt。生产依赖只能通过 Hilt graph、constructor injection、`di/hilt/*` module、`@HiltViewModel`、`@HiltWorker`、`@AndroidEntryPoint` 或 Hilt 提供的 production port 完成。任何第二套 DI 机制、手写 subgraph、legacy compat 旁路、静态 provider callback 都应该视为回归。

不要恢复：

- `ElymBotAppContainer.kt`
- `ElymBotAppContainer.kt`
- `ComposeViewModelHelpers.kt`
- `astrBotViewModel(...)`
- `viewModelFactory`
- `ViewModelProvider.Factory`
- `Default*ViewModelDependencies`
- `MainActivityDependencies`
- global registry / service locator / `*.port` 静态注入
- static `install*` / `configure*` dependency callback
- 在 QQ / plugin / toolsource 热路径里 direct new runtime subgraph
- 用 `Legacy*Adapter` 重新承接 production repository port binding

`RepositoryPortSourceContractTest.kt` 已经切到 `di/AppBootstrapper.kt`，并开始保护：

- `RepositoryPortModule` 不绑定 `Legacy*Adapter`
- `RuntimeServicesModule` 不绑定 legacy compat runtime adapter
- production path 不再使用 static dependency provider callback
- `AppBootstrapper` 不手动 install Phase3 transaction / QQ runtime dependencies
- `AppBootstrapper` 不安装 `RuntimeContextDataRegistry`、`AppBackupDataRegistry`、`ConversationBackupDataRegistry`、`ContainerBridgeStateRegistry`
- `RuntimeContextResolver` 不读 `RuntimeContextDataRegistry`
- core backup/container mainline 不读 registry service locator
- `di/AstrBotViewModelDependencies.kt` 不再声明或承载生产 contracts
- `app-integration` 的 `BackupDataPortAdapter.kt` / `BackupModule.kt` 承载 Hilt backup data port wiring
- `restoreSessionsDurable(...)`
- phase-2 启动不再使用 `runBlocking(Dispatchers.IO)`

`32dd105` / `fb8e7ff` / `d060c5e` / `113dc12` 后这块又新增几组保护：

- `AppStartupChainContractTest`
  - startup chain 文件必须存在
  - `AppStartupRunner.run()` 顺序固定
  - 不允许切回无序 multibinding
- `StrictHiltOnlyFinalContractTest`
  - `AppInitializer` / `InitializationCoordinator` / no-op repository initializer 文件必须保持删除，生产启动主线只看 `AppBootstrapper -> AppStartupRunner -> di/startup/*`
- `AppBootstrapperContractTest`
  - `AppBootstrapper.bootstrap()` 必须保持薄壳
  - 不允许重新手写 assemble startup dependencies
- `StrictHiltOnlyContractTest` / `StrictHiltOnlyFinalContractTest`
  - 明确把 Hilt-only 作为生产 DI 唯一合法路径
- `NoLegacyAdapterContractTest`
  - 禁止把 `Legacy*Adapter` 再写回生产主线
- `NoManualRuntimeSubgraphContractTest`
  - 禁止在 QQ runtime graph 热点 direct new 关键 runtime 子服务
- `NoProductionFileDeprecationSuppressContractTest`
  - 禁止用 `@Suppress("DEPRECATION")` 把生产热点重新接回旧 compat API
- `StrictHiltOnlyStartupHotspotSourceTest`
  - 保护 startup 热点继续保持 Hilt-only

新增测试辅助：

- `app/src/test/java/com/elymbot/android/di/LegacyViewModelDependencyTestContracts.kt`：只给测试保留旧 dependency contract 名称，不是生产事实

### 3.6 Cron 执行记录

Cron 已有执行记录表、coordinator、run-now port、delivery port 和 active capability 管理工具，但 WorkManager 真机行为仍建议手工回归：

- run once 成功后是否删除 job
- recurring 是否重排 nextRunTime
- retryable failure 是否 Result.retry
- delivery summary 是否记录
- pause/resume/list runs/update/run-now 是否经 `CronJobUseCases` / `ActiveCapabilityRuntimeFacade` 保持一致
- scheduled task trigger 下 active capability tools 是否隐藏，避免定时任务自我递归

### 3.8 Backup 覆盖面不足

第 25 期后 backup 生产入口已经退成 Hilt 注入服务：

- `AppBackupService` / `ConversationBackupService` 私有持有实例级 repository
- `AppBackupRepository` / `ConversationBackupRepository` 不再是 production `object`
- backup data ports 来自 `app-integration` 的 `BackupDataPortAdapter.kt` + `BackupModule.kt`

备份测试绿不代表新表都能恢复。当前未覆盖：

- resource center
- cron jobs
- cron execution records
- config context/MCP/skills 历史兼容字段
- plugin config/state

`e495263` 后 `webSearchEnabled`、`proactiveEnabled`、`includeScheduledTaskConversationContext` 已能随 Config backup round-trip，但这不改变 Resource/Cron/plugin 仍未覆盖的事实。

## 4. 推荐手工回归清单

涉及这些模块时建议手工验证：

1. App Chat 普通文本发送，确认流式/非流式回复正常。
2. App Chat 带 prompt skill / tool skill，确认 prompt 注入和 tool descriptor 可见性正确。
3. Resource Center 新增 MCP / Skill，并确认当前 config projection 生效。
4. QQ 真桥消息，确认 OneBot adapter、wake word、回复发送正常。
5. 创建 run once cron，确认执行、投递、执行记录和删除逻辑。
6. 创建 recurring cron，确认失败/retry/重排。
7. 插件启停、恢复 suspension、schema config 页面。
8. App backup 导出/导入，确认不会误以为 Resource/Cron 已恢复。
9. `elymbot://plugin/install` / `elymbot://plugin/repository` deep link，确认会先弹确认框再进入安装。
10. QQ 登录页面在 bridge ready 且二维码为空时，确认会自动触发一次 QR refresh。
11. NapCat/container 首次 warmup 与已有 install 修复路径，确认 bundled / upstream fallback 两种口径都不被误写坏。
12. 开启 Web Search 后分别验证新闻、天气、实时查询，确认 configured provider 与 local fallback diagnostics 可读。
13. 在 App Chat / QQ 里发明显提醒请求，确认模型漏调工具时 host fallback 不会静默承诺失败任务。

## 5. 易错点

- 不要只跑旧 `runtime/plugin` 单测就说插件平台没问题；还要跑 feature/plugin 和 architecture tests。
- 不要只跑 DB schema test 就说备份覆盖没问题。
- 不要把 `RepositoryPortSourceContractTest` 仍当成旧容器测试；当前它已经在保护 `AppBootstrapper` 的真实 wiring。
- 不要把 compat/test bridge 当 production mainline：`PluginRuntime*Provider.installFromHilt(...)`、`PluginExecutionHostApi`、`createCompatPluginHostCapabilityGatewayFactory()` 都不是生产事实。
- 维护背景文档时要写明对应期次的真实状态；第 25 期后 voiceasset/provider runtime/backup 相关事实以当前迁移后的 feature/app-integration/Hilt 入口为准，不要继续沿用旧 root app 路径。
