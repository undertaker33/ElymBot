## 3. 当前路由面

插件相关 route 仍是：

- `plugins`
- `plugins/detail/{pluginId}`
- `plugins/manager`
- `plugins/market/detail/{pluginId}`
- `plugins/detail/{pluginId}/workspace`
- `plugins/detail/{pluginId}/logs`
- `plugins/detail/{pluginId}/triggers`
- `plugins/detail/{pluginId}/config`

`AstrBotAppScaffoldParts.kt` 负责把这些 route 接到 feature plugin presentation 页面。

插件 UI 的 ViewModel 获取方式是 Hilt；`bbb2f5f` 后当前生产绑定是：

- `PluginViewModel`：`@HiltViewModel` + `@Inject constructor(private val bindings: PluginViewModelBindings)`
- `PluginViewModelBindings` / `DefaultPluginViewModelBindings` 在 `feature/plugin/presentation/PluginViewModel.kt`
- `d060c5e` 后 `PluginViewModelBindings` 已拆成 market / config / workspace / governance / management 五组子 binding interface，`DefaultPluginViewModelBindings` 仍是 Hilt 注入的默认实现，但不应继续在 `PluginViewModel` 里内联匿名 controller 或 static repository 入口
- `DefaultPluginViewModelBindings` 由 `di/hilt/ViewModelDependencyModule.kt` 提供
- installer/catalog/subscription/install intent 由 `di/hilt/PluginProvisioningModule.kt` 提供
- `PluginViewModelBindings` 当前还显式暴露：
  - `records`
  - `repositorySources`
  - `catalogEntries`
  - `governanceReadModels`
  - `logBus`

不要再恢复 `astrBotViewModel(...)`、手写 `ViewModelProvider.Factory`，也不要把 `PluginViewModelDependencies` 写成当前生产入口。

另外，`b045602` 起 plugin deep link 安装入口不再是“打开 URI 就直接装”，`bbb2f5f` 仍沿用：

- `MainActivity` 会先把 `astrbot://plugin/install` / `astrbot://plugin/repository` 解析成 `PluginDeepLinkInstallRequest`
- 用户确认后才真正执行安装 intent

这属于入口层，不是 `PluginScreen` 自己处理的逻辑；排查安装跳转时要同时看 `MainActivity.kt` 和 `MainActivityPluginDeepLinkTest.kt`。

## 4. UI 与 domain/use case 的关系

当前插件管理路径已经抽出：

- `PluginManagementUseCases`
- `PluginPresentationController`

`PluginViewModel` 的 enable / disable / uninstall / recover / suspend 等管理动作应通过 controller/use case，不要直接绕回 repository/runtime 细节。

## 5. UI 与运行时的关系

插件 UI 不是真源。

它展示/操作的真源来自：

- `FeaturePluginRepository`
- plugin domain ports/use cases
- plugin runtime state/log bus
- plugin config snapshots
- plugin governance state

`fb8e7ff` 下 UI 详情页 / 配置页 / workspace 还要额外记住：

- detail 管理可用性与只读门控不只是看 `record.enabled`
- `PluginGovernanceMutationGate`、`PluginFailureUiState`、`PluginUpdateAvailability`、`PluginGovernanceReadModel`
  才是当前 detail action / read-only 行为的主要依据
- workspace/config snapshot 通过：
  - `resolvePluginWorkspaceSnapshot(...)`
  - `resolvePluginConfigSnapshot(...)`
  - `savePluginCoreConfig(...)`
  - `savePluginExtensionConfig(...)`
  走 `PluginConfigBindings`
- market source / install intent / catalog entries 走 `PluginMarketBindings`
- workspace import/delete/snapshot 走 `PluginWorkspaceBindings`
- governance read/mutation gate 走 `PluginGovernanceBindings`
- enable/disable/uninstall/recover/suspend 走 `PluginManagementBindings`

ToolSource、MCP、WebSearch、Active Capability 的真实规则在 `feature/plugin/runtime/toolsource/*`，UI 只做展示与入口。

## 8. 易错点

- 新插件 UI 不应回到 root `ui/plugin`。
- 插件管理动作不要绕过 `PluginPresentationController`。
- 市场详情和本地详情 route 仍是不同语义，不要混用 ID。
- UI 文档不要重新定义 ToolSource 规则；运行时规则看 07。
- 插件 UI 当前通过 Hilt 拿 `PluginViewModelBindings`；不要把旧 container/viewModel helper 或 `PluginViewModelDependencies` 当生产入口。
- 不要把 `PluginViewModel` 写回巨型 inline dependency assembler；`d060c5e` 后 binding 文件按 market/config/workspace/governance/management 拆开，并有 contract test 保护。
- detail/config/workspace 的只读/按钮门控当前要同时看 governance/failure/update 状态；不要只看 `PluginInstallRecord.enabled`。
