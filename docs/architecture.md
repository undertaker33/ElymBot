# AstrBot Android 原生应用架构

## 项目目标

本项目的目标是构建一个**原生 Android 应用**，以 AstrBot 的产品能力为中心，同时保持 NapCat 作为专门处理 QQ 协议的运行时边界。应用层面负责提供配置界面、插件和知识库管理等用户交互；底层的 NapCat 保持轻量且专注，只处理 QQ 协议及其防检测逻辑，从而保证架构清晰、职责单一。

## 模块边界

### 原生应用（Native App）

原生应用负责与用户交互和配置管理：

- **机器人配置与编排：** 管理多个 bot 的运行状态、参数设置和流程编排。  
- **Provider 管理：** 负责聊天、文本转语音（TTS）、自动语音识别（ASR）等模型提供商的配置、创建、更新和删除。  
- **人格管理：** 支持为不同会话创建并编辑虚拟人格（persona），以控制机器人的语调和行为。  
- **会话上下文管理：** 对每个对话的历史、状态和记忆进行本地持久化，以便持续对话。  
- **平台日志中心：** 为运行时和系统提供统一的日志查看和过滤界面。  
- **插件与知识库管理界面：** 管理扩展插件和知识库，提供安装、更新和权限配置 UI。  

### 容器运行时（Container Runtime）

容器层封装了 NapCat 运行时，专门处理 QQ 协议和敏感行为：

- **NapCat 运行时：** 实现 QQ 协议的事件收发、状态保持以及协议级别逻辑。  
- **QQ 事件入口与出口：** 处理 QQ 消息的接入和发送，与原生应用通过桥接服务交互。  
- **反检测敏感行为：** 针对平台的反自动化检测，封装相应的风控策略和运行时行为。  

## 运行时结构

运行时按从 UI 到协议的链路划分，整体流程如下：

```
Android UI → 原生领域层 → 本地存储 → 容器桥接服务 → NapCat 运行时 → QQ 平台
```

用户通过原生界面进行操作，领域层处理业务逻辑并与本地存储交互。容器桥接服务负责在原生与 NapCat 之间传递消息和状态，NapCat 进一步将消息发送给 QQ 平台，并接收平台事件反馈。

## 阶段性计划

为了降低复杂度，本项目按阶段推进：

### 第 1 阶段

1. 创建以 Android SDK 34 为基础的原生工程。  
2. 建立应用导航和应用框架。  
3. 定义运行时状态模型及桥接服务的占位接口。  
4. 设计本地设置页面及日志中心。  

### 第 2 阶段

1. 实现 QQ 机器人模块界面。  
2. 完成 provider 的增删改查功能。  
3. 完成人格（Persona）的增删改查功能。  
4. 实现会话上下文的持久化保存和恢复。  

### 第 3 阶段

1. 将桥接服务接入真实的 NapCat 运行时。  
2. 增加健康检查与状态同步机制。  
3. 打通 QQ 消息的发送与接收流程，完成编排逻辑。  

### 第 4 阶段

1. 实现插件管理界面及其运行机制。  
2. 完成知识库的创建、导入、同步与管理功能。  
3. 提供工具注册与执行策略管理，支持扩展第三方工具。  

## Android 兼容性要求

为保证兼容性和合规性，开发过程中需注意以下事项：

1. **编译与目标版本：** `compileSdk = 34`，`targetSdk = 34`；需及时跟进新版本的特性与限制。  
2. **避免隐藏 API：** 不应调用隐藏的系统 API，也不要依赖旧版后台任务或广播机制。  
3. **前台运行行为：** Android 14 起前台服务、通知权限等政策更严格，须遵守系统限制。  
4. **存储与通知策略：** 在 Android 15 及以后版本，应确保文件访问和通知行为符合最新政策。  

## 总结

通过上述架构设计，本项目将在原生应用与容器运行时之间建立明确边界。原生应用负责界面与配置，容器运行时专注 QQ 协议及防检测，实现职责分离。逐步推进的阶段计划确保复杂功能逐层落地，最终使 AstrBot 能在 Android 平台上稳定提供丰富的聊天与插件功能。
